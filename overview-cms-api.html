API 참조

API 레퍼런스도 참조하십시오 .

일반 정보

기본 URL

의 기본 URL CMS API이다:

        https://cms.api.brightcove.com/v1

계정 경로

모든 경우에 특정 요청이 이루어집니다. 비디오 클라우드계정. 따라서 항상 용어accounts뒤에 계정 ID를 기본 URL에 추가합니다.

        https://cms.api.brightcove.com/v1/accounts/{account_id}

입증

요청에 대한 인증에는 다음이 필요합니다. 승인 헤더 :

        Authorization: Bearer {access_token}

NS access_token에서 가져와야 하는 임시 OAuth2 액세스 토큰입니다. 브라이트코브 OAuth서비스. 자세한 내용은브라이트코브 OAuth 개요 .

속도 제한

이것 CMS API데이터에 대한 캐시되지 않은 읽기 액세스를 제공합니다. 이는 시간에 민감한 게시 워크플로에 중요합니다. CMS API데이터를 빠르게 읽고 그것이 정확한지 확인합니다.

NS CMS API대규모 런타임 사용(예: 트래픽이 많은 공개 웹 페이지에서 비디오에 액세스)에는 적합하지 않습니다. 트래픽이 많은 애플리케이션의 경우 다음과 같은 캐시된 인터페이스를 사용해야 합니다. 재생 API , 갤러리, 플레이어 또는 기본 SDK.

의 성능을 보장하기 위해비디오 클라우드시스템, 20개 이하의 동시 호출 CMS API계정당 허용됩니다. 액세스 빈도는 초당 10개 미만의 요청이어야 합니다.

동시 호출은 병렬로 실행되는 여러 프로세스 또는 앱이 API 요청을 할 때 발생합니다. 요청을 완료하는 데 1초 이상 걸릴 수 있으므로 초당 10개 이상의 요청을 만들지 않더라도 동시성 제한을 초과할 수 있습니다.

여러 응용 프로그램을 통합할 경우 CMS API계정의 경우 이러한 애플리케이션에는 동시성 제한 또는 속도 제한에 도달한 인스턴스를 처리하기 위해 백오프 및 재시도 논리가 있어야 합니다.

비율 또는 동시성 제한을 초과하는 경우429 - TOO_MANY_REQUESTS오류가 반환됩니다.

참조 ID 충돌

참조 ID의 고유성을 보장하기 위해 CMS API할당된 비디오에 대한 작업 후 최대 3분 동안 ID를 잠급니다. 너무 빨리 실패한 요청을 다시 시도하거나 이전에 할당된 비디오를 삭제한 후 너무 빨리 참조 ID를 재사용하려고 할 때 409 오류가 반환될 수 있습니다. 참조오류 메시지 참조자세한 사항은.

동영상 자산 한도

동영상당 자산은 1,000개로 제한됩니다. 자산에는 디지털 마스터뿐만 아니라 변환, 오디오 트랙, 텍스트 트랙, 이미지가 포함됩니다. 변환 및 이미지의 총합이 10-15개를 초과하는 경우는 거의 없으므로 150개 언어에 대해 별도의 오디오 트랙과 텍스트 트랙이 있더라도 여전히 350개 미만의 자산을 갖게 됩니다.

사용상의 주의

NS CMS API통합 및 게시 워크플로용입니다. 공개 웹 페이지에서 이 API에 대한 클라이언트 측 호출을 할 수 없습니다. 그렇게 하면 API에 대한 클라이언트 자격 증명이 노출되기 때문입니다(이러한 이유로 API는 CORS를 사용할 수 없음). 서버 측 프록시를 통해 API 호출을 라우팅하는 경우 클라이언트 측(웹) 앱에서 호출할 수 있습니다. 브라이트코브 학습 서비스여러 제공샘플 앱이 접근 방식을 사용하는이런 종류의 앱을 만들기 위한 가이드 .

모든 비디오 또는 대규모 비디오 세트를 검색하려는 경우 다음을 확인하십시오. 대용량 데이터 세트에 대한 참고 사항 .

하드 코딩된 URL 피하기

썸네일, 포스터, 비디오 파일 및 기타 미디어의 URL은 페이지나 애플리케이션에 하드 코딩되어서는 안 됩니다. NS CMS API는 항상 미디어 파일의 현재 URL을 반환하지만 URL 자체는 변경될 수 있습니다. 당신은 사용해야합니다 CMS API ( 또는재생 API ) 페이지가 로드될 때마다 이러한 URL을 검색하거나 6시간 이하 동안 캐시하도록 요청합니다.

비디오 및 이미지 URL 캐싱

페이지 성능을 향상시키기 위해 비디오 및 이미지의 URL을 캐시할 수 있지만 캐시는 정기적으로 새로 고쳐야 합니다. 페이지 성능을 향상시키기 위해 검색한 URL을 캐시하는 경우 6시간에 한 번 이상 API 호출을 반복하여 캐시를 새로 고쳐야 합니다.

동영상 검색

브라이트코브의 CMS API에서 동영상을 검색하는 프로그래밍 방식을 제공합니다. 비디오 클라우드도서관.

비디오 데이터에 대한 기본 및 복잡한 검색을 수행하려면 다음을 사용합니다. q매개변수:

        https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q={search terms}

동영상 검색 방법에 대한 자세한 내용은동영상 검색문서.

재생 목록의 경우 검색 문자열에 대해 지원되는 값이 더 제한됩니다. 현재 다음으로 검색할 수 있습니다. type , name , description , 그리고reference_id . 다음은 유효한 검색의 예입니다.

  • q=type:EXPLICIT
  • q=type:ACTIVATED_OLDEST_TO_NEWEST
  • q=type:ALPHABETICAL
  • q=bears+otters (이름, 설명 또는 참조 ID에는 "곰" 또는 "수달"이 포함되어야 함)
  • q=%2Bname:bears+type:EXPLICIT (이름에는 "곰"이 포함되어야 함)

보다재생 목록 검색자세한 사항은.

검색어 참고playable:true/false재생 가능한(또는 그렇지 않은) 동영상만 반환하는 데 사용할 수 있습니다. 
        https://cms.api.brightcove.com/v1/accounts/:account_id/videos?q=%2Bplayable:true

이 쿼리를 사용하려면 반환된 비디오가 재생 가능해야 합니다(비활성화되지 않고, 예약되지 않고, 재생 가능한 변환이 부족하지 않아야 함).

있는 동영상을 검색하는 경우 ~ 아니다재생 가능, 사용q=%2Bplayable:false .

페이징 결과

사용 limit 요청 시 반환할 항목 수를 지정하는 매개변수(최대 100개). 그런 다음 사용할 수 있습니다. offset 보다 큰 결과 세트를 통해 페이지에 대한 매개변수limit . NS offset건너뛸 항목의 수입니다.

예를 들어, 다음 검색은 총 결과 세트에 최소 75개의 비디오가 있다고 가정할 때 총 결과 세트의 비디오 51-75를 반환합니다.

        /videos?q=updated_at:2014-01-01..2014-06-30&limit=25&offset=50

NS limit그리고offset매개변수는 동영상과 재생목록 모두에 사용할 수 있습니다.

메모: limit그리고offset ~하다 ~ 아니다재생 목록에 속한 비디오를 요청할 때 작동합니다. 이 요청은 항상 재생 목록의 모든 비디오를 반환합니다.

페이징 모범 사례

CMS API에서 동시 수정 작업이 진행 중일 수 있으므로 결과 집합을 페이징할 때 다음 단계를 따르는 것이 좋습니다.

  1. 을 만들다 count 결과 세트의 총 비디오 수를 가져오도록 요청하십시오. 
          /accounts/578380111111/counts/videos?q=tags:nature
  2. 사용 limit 그리고 offset 매개변수를 사용하여 결과 집합에서 데이터 그룹을 반환합니다. 
          /accounts/578380111111/videos?q=tags:nature&limit=20&offset=50
  3. 일부 페이지에는 20개 미만의 동영상이 있을 수 있습니다. 처음과 같이 많은 비디오를 요청했을 때 결과 세트의 끝에 도달했음을 알 수 있습니다. count요구.

요약하자면, 비디오 수가 원본과 같아질 때까지 페이지를 계속 검색하십시오. count이 숫자는 과대평가 측면에서 오류가 발생해야 하기 때문에 요청합니다. 요청:

      count / page-size + 1 page

비디오 결과 정렬

매개변수 사용sort=field_name결과 정렬 방법을 지정하려면 비디오와 재생 목록을 모두 정렬할 수 있습니다. 다음 비디오 필드를 기준으로 정렬할 수 있습니다. [1-1]

내림차순으로 필드 이름 앞에 빼기 기호 ( sort= -field_name ) 를 붙입니다.
  • name
  • reference_id
  • created_at
  • published_at
  • updated_at
  • schedule_starts_at (참고: 이것은종류필드 - 검색 필드는schedule.starts_at )
  • schedule_ends_at (참고: 이것은종류필드 - 검색 필드는schedule.ends_at )
  • state
  • plays_total [1-2]
  • plays_trailing_week [1-2]

노트

  • [1-1] 화상 검색 호출에 대한 정렬 값을 제공하지 않으면 결과가 관련성을 기준으로 정렬됩니다. 정렬 값을 제공하지 않으면GET화상 통화, 결과는 다음 기준으로 정렬됩니다. updated_at내림차순.
  • [1-2] 정렬할 수 있습니다. plays_total또는plays_trailing_week , 하지만 이러한 필드는 결과에 포함되지 않습니다.

  • name
  • updated_at (기본)

모든 비디오 및 대용량 데이터 세트

계정에 있는 모든 동영상을 검색하거나 많은 수의 동영상을 검색하는 경우 다음 사항에 유의해야 합니다.

  1. 허용된 가장 큰 값을 사용하고 싶을 수 있습니다. limit (100), 그러나 API 요청 시간 초과 가능성을 최소화하려면 25개 이하의 배치로 비디오를 검색하는 것이 좋습니다.
  2. 큰 데이터 세트를 페이징할 때 작업 중에 비디오 데이터가 업데이트되어 응답에서 항목이 이동할 수 있습니다.
    • 연속 페이지에서 항목이 반복되는 것을 볼 수 있습니다.
    • 항목이 이전 응답 세트로 이동되었기 때문에 누락될 수 있습니다.

    첫 번째 가능성을 고려하려면 동영상 검색을 완료한 후 앱에서 전체 항목 목록의 중복을 제거해야 합니다. 두 번째 가능성을 처리하려면 검색된 총 항목 수(중복 제거 후)를 예상한 수와 비교한 다음 요청을 다시 실행하여 결과를 last_modified_date(내림차순)로 정렬해야 합니다. 놓친 항목을 선택하기 위해 둘 이상의 배치를 검색합니다.

  3. 반환된 결과를 적절하게 정렬하여 이전 항목의 시나리오 가능성을 줄일 수 있습니다. 기본 정렬 기준관련성검색은 키워드, 태그 및 사용자 정의 필드 값의 조합을 찾는 복잡한 알고리즘을 기반으로 합니다. 여러 키워드, 태그 및/또는 사용자 정의 필드를 기반으로 비디오를 검색하는 경우 관련성을 기준으로 정렬하는 것이 정확히 원하는 것입니다. 그러나 전체 또는 많은 비디오 세트를 검색하려는 경우에는sort매개변수를 명시적으로 지정하면 반환된 항목의 순서를 더 잘 제어할 수 있습니다.

비디오 작업

비디오 작업에는 다음이 포함됩니다.

  • 동영상 수 또는 검색 결과 가져오기
  • 모든 비디오 가져오기
  • ID 또는 참조 ID로 하나 이상의 동영상 가져오기
  • 비디오 생성, 검색, 업데이트 및 삭제

  • WebVTT 텍스트 트랙 또는 DFXP 캡션 추가, 업데이트 또는 제거

알림

다음과 같은 경우 알림을 받을 수 있습니다. video_change이벤트는 비디오 라이브러리에서 발생합니다. 알림은 처리할 수 있는 응용 프로그램을 가리켜야 하는 지정한 URL로 전송됩니다. HTTP POST요청.

참고 사항:
  • 비디오 메타데이터가 변경되면video_change이벤트. 동영상에 사용된 자산에 대한 변경 사항은 ~ 아니다변경 이벤트를 트리거합니다.
  • 알림 외에도상태 API비디오와 관련된 수집(수집, 교체 또는 재변환) 작업의 상태를 확인할 수 있습니다.

알림 실패

알림 시스템은 고객 서버의 4xx 또는 5xx 반환을 재시도 가능한 오류로 처리합니다. 실패한 콜백은 최대 20번까지 재시도되며 후속 콜백 사이의 지연은 기하급수적으로 증가합니다. 처음 몇 번의 재시도는 초기 콜백 시도 후 몇 분 이내에 발생합니다. 콜백이 계속 실패하고 20번째 재시도까지 계속되면 재시도 지연은 며칠이 될 것입니다.

video_change이벤트는 자동화된 시스템 프로세스 및 사용자 작업에 의해 트리거되므로 이러한 이벤트는 Studio 또는 API를 통해 변경한 동영상 속성과 항상 일치하지는 않습니다.
